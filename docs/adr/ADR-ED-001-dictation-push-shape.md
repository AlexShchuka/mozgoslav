# ADR-ED-001 — Single dictation push endpoint, raw octet-stream

## Status

Accepted, 2026-04-28. Closes issue #60.

## Context

Two dictation push endpoints coexisted in
`backend/src/Mozgoslav.Api/Endpoints/InternalEndpoints.cs`:

| Path | Body | Caller |
|---|---|---|
| `POST /api/dictation/push/{sessionId:guid}` | JSON `{samples: float[], sampleRate, offsetSeconds}` | `frontend/electron/dictation/DictationOrchestrator.ts` only |
| `POST /api/dictation/{sessionId:guid}/push` | raw `application/octet-stream` (auto-detects WebM/Opus, Ogg, or `audio/pcm` Float32 LE) | `frontend/src/api/dictationPush.ts` (renderer) and `DictationOrchestrator.ts` after migration |

Both paths route into the same `IDictationSessionManager` —
`PushAudioAsync(AudioChunk)` for parsed PCM, `PushRawChunkAsync(byte[])`
for compressed payloads that the server demuxes via ffmpeg. The
duplication produced two integration-test suites
(`InternalDictationPushTests` for the JSON path,
`DictationPushWebmOpusTests` for the raw path) and two divergent
client-side codepaths.

## Decision

**Winner**: `POST /api/dictation/{sessionId:guid}/push` (raw
`application/octet-stream`, with `audio/pcm` content-type signaling
client-encoded Float32 little-endian samples).

**Loser, retired**: `POST /api/dictation/push/{sessionId:guid}`
(JSON-samples shape) and the corresponding `PushChunkRequest` record.

**Justification**:

1. The renderer's binary path (`dictationPush.ts`) is the canonical
   production traffic — browser-recorded WebM/Opus chunks. It is also
   the path covered by the WebM/Opus fixture-driven integration test,
   which exercises chunked MediaRecorder continuation clusters
   end-to-end through ffmpeg.
2. Bandwidth: a 1-second 16 kHz Float32 frame is 64 kB raw PCM versus
   ~1 MB of JSON-encoded floats (≈14 bytes per sample). On a tight
   loopback this is irrelevant; over Syncthing-tunnelled gRPC contours
   it becomes a measurable per-session overhead.
3. The native helper produces Float32 samples at the configured sample
   rate; encoding them to a `Float32Array` byte view is a single
   `Buffer.writeFloatLE` loop in Electron and a `Buffer.BlockCopy` on
   the server. Per-chunk JSON parse/serialise on a 16 kHz audio stream
   is wasted CPU on both sides.
4. The server already auto-detects three content shapes on the raw
   path (WebM, Ogg, `audio/pcm`). Adding a JSON branch on the same
   route would replicate the schism inside one handler; keeping two
   routes replicates it across the API surface. Either way, the choice
   is to collapse the surface — and the raw-stream path is the more
   general of the two.

## Consequences

* `frontend/electron/dictation/DictationOrchestrator.ts`'s
  `pushAudioToBackend` writes a `Float32Array` view of the helper's
  samples into a `Buffer` and posts it with `content-type: audio/pcm`
  to the winning endpoint.
* `backend/tests/Mozgoslav.Tests.Integration/InternalDictationPushTests.cs`
  is archived under `.archive/backend/tests/` (covered the loser path).
  A replacement test
  `InternalDictationPushRawPcmTests.cs` covers the winner path's
  unknown-session 404, empty-payload 400, and happy-path 200 cases.
* `backend/openapi.json` will be regenerated by the next build that
  exposes Swashbuckle; the loser route disappears automatically.
* `backend/src/Mozgoslav.Api/Endpoints/InternalEndpoints.cs` no longer
  declares the nested `PushChunkRequest` record — one fewer
  contract-shaped type to keep in sync with the renderer.
* The `DictationPushWebmOpusTests` suite continues to exercise the
  WebM/Opus branch of the same endpoint without modification.

## Alternatives rejected

| Option | Reason |
|---|---|
| Keep JSON-samples shape as the winner | Renderer already uses raw octet-stream; switching back would require a JS-side WebM-to-Float32 ffmpeg pipeline in the browser sandbox — impossible without a native sidecar that already exists server-side. |
| Keep both endpoints behind a feature flag | Two test suites, two client paths, two openapi entries — the schism that #60 was opened to retire. |
| Move dictation push to gRPC streaming | Disproportionate; the dictation traffic is loopback-only and gRPC adds a build-time codegen step on every contract change. Re-evaluate if Pillar 3's MCP/gRPC stack pulls this naturally. |
| Add `application/json` as a third content branch on the raw endpoint | Reproduces the JSON path inside the handler — same surface area, less honest URL. |
